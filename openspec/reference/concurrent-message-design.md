# Agent 工具执行期间新消息处理：设计演进与 Trade-off

> 设计推演文档。记录了从延迟绑定 → 抢占式调度 → 信息透传（当前 Absorb-Replan）的演进过程。
> 具体实现细节（如 Redis Stream、端侧乐观锁）已不适用于当前架构，但设计推理和 trade-off 分析仍有参考价值。
> 当前实现见：`changes/archive/2026-03-05-session-message-absorb-replan/`

## 一、问题背景

### 1.1 业务场景

**产品形态**：IM 场景的 AI Agent，部署在飞书和企微群聊中。

**技术架构**：通过 RPA 方式控制微信/飞书客户端。

**关键约束**：
- 读取消息需要时间（RPA 从客户端抓取）
- 发送消息需要时间（RPA 控制客户端发送）
- Agent 推理需要时间（LLM 调用）
- 整个链路：用户发消息 → 端侧上报 → 服务端处理 → Agent 推理 → 服务端下发 → 端侧执行

### 1.2 核心问题

在上述链路的任何阶段，用户都可能发新消息。

**问题**：Agent 在处理过程中，如果用户发了新消息，应该怎么办？

**具体场景**：
1. Agent 正在推理，用户发了"算了不用了"
2. Agent 决定发消息，还没执行，用户说"等等，别发"
3. Agent 调用搜索工具，执行期间用户补充了"记得加上日期"
4. 端侧正在执行发送，服务端又收到了新消息

### 1.3 为什么这是一个难题

**传统 chatbot 不存在这个问题**：
- 同步模型：用户发消息 → bot 回复 → 用户再发
- 整个过程串行，不存在"处理中来新消息"

**Agent 的特殊性**：
- 有工具调用（tool call），这是异步阶段
- 工具执行可能耗时几秒甚至更长
- 用户不会干等着

**IM 场景的特殊性**：
- 用户随时可能发消息
- 群聊中其他人也可能发消息
- 消息发出去无法撤回（不可逆）

## 二、设计目标与约束

### 2.1 设计目标

1. **正确性**：Agent 的行为应该符合用户当前的意图
2. **及时性**：能感知用户意图的变化
3. **安全性**：不可逆操作（发消息）要谨慎
4. **简单性**：架构不要太复杂

### 2.2 约束条件

1. **性能约束**：整个链路有延迟，无法做到"实时"
2. **架构约束**：服务端和端侧是分离的，通过消息通信
3. **能力约束**："判断用户意图是否变更"本身是一个难题

---

## 三、三种设计模型

### 3.1 模型一：延迟绑定（Late Binding）

#### 核心思路

把"是否执行"的决策推迟到最后时刻。

#### 实现方式

```
Agent 完成推理
    ↓
决定发送消息（但不立即发）
    ↓
通知端侧
    ↓
端侧重新拉取最新消息列表
    ↓
端侧判断：用户意图是否变更？
    ├─ 是 → 调整或取消发送
    └─ 否 → 执行发送
```

#### 决策者

端侧。

#### 优点

- 能感知到最新状态（端侧重新拉取）
- 理论上最准确（在最后时刻判断）

#### 缺点

- **耦合严重**：Agent 逻辑和端侧判断逻辑耦合
  - Agent 需要告诉端侧"我要发什么"
  - 端侧需要理解"什么情况算意图变更"
  - 改一边就要改另一边
- **判断困难**："意图变更"的判断本身很难
  - 用户说"再加一句"算变更吗？
  - 用户说"帮我改个措辞"呢？
  - 需要 NLU 能力，但端侧通常没有

#### 放弃原因

耦合太重，维护困难。每次改 Agent 逻辑都要同步改端侧。

---

### 3.2 模型二：抢占式调度 + 乐观锁

#### 核心思路

有新消息就放弃旧的，从头来过。端侧用版本号兜底。

#### 实现方式

**服务端（悲观策略 / 抢占式调度）**：

```
Agent 开始处理
    ↓
处理过程中持续检查：有新消息吗？
    ├─ 有 → 放弃当前中间结果，用新消息重新触发 Agent
    └─ 无 → 继续处理
    ↓
处理完成，下发指令给端侧
```

**端侧（乐观锁）**：

```
收到服务端指令（带版本号）
    ↓
校验版本号：和当前消息状态一致吗？
    ├─ 一致 → 执行
    └─ 不一致（过期）→ 不执行
```

**版本号机制**：
- 版本号由端侧生成
- 基于当前手机端的消息记录
- 服务端处理时记录版本号，下发指令时带上
- 端侧执行前校验：版本号是否过期

#### 决策者

- 服务端：决定是否放弃中间结果
- 端侧：决定是否执行指令（版本号校验）

#### 为什么是"双保险"

1. **服务端能检测到的**：处理过程中来的新消息 → 放弃重来
2. **服务端漏掉的**：服务端检查完到端侧执行之间来的新消息 → 端侧版本号拦住

#### 优点

- **解耦**：服务端和端侧各管各的
  - 服务端只管"有新就放弃"
  - 端侧只管"版本号校验"
  - 不需要共享"意图变更"的判断逻辑
- **双保险**：两层防护，更安全

#### 缺点

- **浪费计算**：中间结果全扔了
  - 即使用户只是补充信息，之前的推理也白费了
- **延迟增加**：放弃重来意味着用户等更久
- **架构复杂**：
  - 服务端要维护"正在处理"的状态
  - 端侧要维护版本号
  - 两边都有状态，调试困难

#### 适用条件

Agent 耗时长，新消息出现概率高。

在这种情况下，"放弃重来"虽然浪费，但能保证正确性。

---

### 3.3 模型三：信息透传 + Agent 自决（当前）

#### 核心思路

系统只负责传递信息，决策交给 Agent。

#### 实现方式

**Peek 机制**：

```
工具执行前/后
    ↓
"偷看"消息队列：有新消息吗？（不消费，不 ACK）
    ↓
根据结果决定下一步
```

**工具执行后的处理**：

```
工具执行完毕
    ↓
Peek：有新消息吗？
    ├─ 无 → 正常返回工具结果
    └─ 有 → 按时序拼接上下文
```

**上下文拼接格式**：

```
1. [工具执行中...]          ← 占位，表示 Agent 确实调用了工具
2. 在工具执行期间存在新事件: xxx  ← 新消息
3. 工具执行完成: ...         ← 真实的工具结果
```

**发送类工具的特殊处理**：

```
准备执行 send_message
    ↓
Peek：有新消息吗？
    ├─ 有 → 不执行，返回"发送中止，请重新评估"
    └─ 无 → 正常执行
```

**端侧**：

无特殊逻辑。收到指令就执行。

#### 决策者

Agent。

系统把所有信息（工具结果 + 新消息）都给 Agent，Agent 自己决定怎么响应。

#### 为什么取消了乐观锁

因为 Agent 足够快了。

- 处理窗口短 → 服务端能检测到大部分新消息
- 端侧的乐观锁变成了多余的复杂度

#### 优点

- **简单**：系统只做信息透传，不做复杂判断
- **解耦**：端侧无逻辑，服务端不替 Agent 决策
- **灵活**：不同情况由 Agent 自己判断
- **保留中间结果**：工具结果不扔掉，可能还有用

#### 缺点

- **可能过度中止**：发送类工具只要检测到新消息就中止
  - 用户只是补充信息，也会触发中止
  - 这是"错杀"
- **依赖 Agent 能力**：Agent 需要理解"工具执行期间有新消息"这种上下文格式
- **依赖性能**：如果 Agent 变慢，新消息概率上升，当前设计可能不够用

#### 适用条件

1. Agent 处理足够快（新消息出现概率低）
2. 中止的成本低（多一次 LLM 调用，可接受）
3. 错误发送的成本高（消息无法撤回）

## 四、演进的关键洞察

### 4.1 性能影响架构

| 状态 | Agent 耗时 | 新消息概率 | 需要的机制复杂度 |
|------|-----------|-----------|----------------|
| 之前 | 长 | 高 | 高（双保险） |
| 现在 | 短（缩减一半） | 低 | 低（单层透传） |

**洞察**：性能提升 → 处理窗口缩短 → 问题发生概率降低 → 可以简化架构。

这是**用性能换架构简化**的典型案例。

### 4.2 决策权的转移

| 版本 | 谁判断意图变更 | 谁决定是否执行 |
|------|--------------|--------------|
| 模型一 | 端侧 | 端侧 |
| 模型二 | 系统（有新就算变更） | 系统 + 端侧 |
| 模型三 | Agent | Agent |

**洞察**：从"系统/端侧替 Agent 决策"到"信任 Agent 自己决策"。

背后的假设变了：
- 之前：系统/端侧比 Agent 更擅长判断意图变更
- 现在：Agent（LLM）足够聪明，能自己判断

### 4.3 简化的代价

当前设计是"保守策略"：有新消息就中止发送类工具。

这会导致"错杀"：用户只是补充信息，也会触发中止。

**为什么可接受**：
- Agent 足够快，错杀次数少
- 错杀代价小（多一次 LLM 调用）
- 放过代价大（消息发出去无法撤回）

**权衡公式**：`错杀次数 × 错杀代价 < 放过次数 × 放过代价`

在当前性能条件下，这个不等式成立。

## 五、未采用的方案

### 5.1 基于意图分类的混合策略

#### 思路

新消息进来时，先用轻量级判断分类意图：

```
新消息进来
    ↓
轻量级判断（规则 / 微型模型）
    ↓
意图分类
    ├─ "中断/取消" → 抢占式（放弃旧结果）
    └─ "补充/闲聊" → 延迟绑定（保留旧结果）
```

#### 优点

- 理论上最精确
- 减少错杀（补充信息不会触发中止）

#### 缺点

- 需要引入"路由层"
- 意图分类本身很难（很多消息意图模糊）
- 增加复杂度

#### 未采用原因

当前方案够用，不值得增加复杂度。

如果未来 Agent 变慢、错杀率上升，可以考虑引入。

## 六、当前设计的详细实现

### 6.1 Peek 机制

**技术选型**：Redis Stream

**实现**：
- `XPENDING`：检查有无未消费的消息
- `XLEN`：检查队列长度
- 不执行 `XREADGROUP`（那会消费消息）

**特点**：只检查，不消费。

**用途**：
1. 工具执行前：判断是否中止发送类工具
2. 工具执行后：判断是否需要拼接新消息

### 6.2 发送类工具的中止逻辑

**配置**：`toolsToAbortOnPendingEvents: ['send_message']`

**逻辑**：

```
准备执行工具调用列表
    ↓
分类：哪些是发送类工具？哪些不是？
    ↓
Peek：有新消息吗？
    ├─ 无 → 全部正常执行
    └─ 有 → 发送类工具中止，其他工具正常执行
```

**中止的返回值**：

```
【发送中止】消息"xxx"未发送，用户没有收到。请重新发送。
```

标记为 `is_error: true`，让 Agent 知道这是异常情况。

### 6.3 上下文拼接逻辑

**触发条件**：工具执行后 peek 发现有新消息

**拼接顺序**：

1. **假的工具结果**：`[工具正在执行中...]`
   - 作用：占位，满足 API 格式要求（每个 tool_call 必须有对应的 tool_result）
   - 同时预存 `_meta`（如 `sentMessageIds`，用于消息撤回）

2. **新消息**：`在工具执行期间存在新事件: xxx`
   - 格式化后插入
   - 让 Agent 知道这是"执行期间"发生的

3. **真的工具结果**：`工具执行完成 (tool_use_id: xxx): ...`
   - 作为 event 类型存储（不是 tool 类型）
   - 因为前面已经有一个假的 tool 结果了

**为什么这样设计**：

- 反映真实的时序：Agent 调用工具 → 用户发消息 → 工具返回结果
- Agent 能正确理解上下文
- 满足 API 格式要求（Claude API 要求 tool_call 和 tool_result 配对）

### 6.4 持久化一致性

**存储顺序**：

1. 假工具结果存库（带 `_meta`）
2. Fetch 新消息（fetch 时自动存库）
3. 真工具结果存库（作为 event 类型）

**为什么假工具结果也要存**：

- 需要预存 `_meta`（如 `sentMessageIds`）
- 后续真工具结果存为 event 类型，不会覆盖 `_meta`

## 七、Trade-off 汇总

### 7.1 三种模型的 Trade-off

| 维度 | 模型一（延迟绑定） | 模型二（抢占式+乐观锁） | 模型三（信息透传） |
|------|------------------|---------------------|-----------------|
| 复杂度 | 高（耦合） | 高（双状态） | 低 |
| 计算浪费 | 低 | 高（放弃重来） | 低 |
| 延迟 | 高（端侧判断） | 高（重来） | 低 |
| 准确性 | 高（理论上） | 中 | 依赖 Agent |
| 适用条件 | 端侧有判断能力 | Agent 慢 | Agent 快 |

### 7.2 当前设计内部的 Trade-off

**保守 vs 激进**：
- 当前：保守（有新消息就中止）
- 激进：只有"取消"意图才中止
- 选择保守的原因：错杀代价 < 放过代价

**信任系统 vs 信任 Agent**：
- 当前：信任 Agent（系统只透传）
- 替代：系统判断意图（引入路由层）
- 选择信任 Agent 的原因：简单，且 LLM 能力足够

**单层防护 vs 多层防护**：
- 当前：单层（服务端）
- 之前：双层（服务端 + 端侧乐观锁）
- 选择单层的原因：Agent 足够快，单层够用

## 八、设计的前提条件与风险

### 8.1 前提条件

1. **Agent 足够快**：处理窗口短，新消息出现概率低
2. **中止成本低**：多一次 LLM 调用，可接受
3. **错误发送成本高**：消息无法撤回

### 8.2 风险与应对

| 风险 | 触发条件 | 应对策略 |
|------|---------|---------|
| 错杀率上升 | Agent 变慢 | 引入意图分类路由层 |
| 中止成本上升 | LLM 成本大幅上涨 | 接受更多"冒险发送" |
| Agent 不理解上下文格式 | 换模型 | 调整 system prompt / 拼接格式 |

## 九、与通用概念的关系

### 9.1 与"快速失败"的关系

**快速失败（Fail Fast）**：
- 核心：尽早检测错误，尽早报告
- 当前设计：检测到"可能的风险"（新消息）→ 中止

**相似点**：
- 都是"早发现，早处理"
- 都是保守策略

**不同点**：
- 快速失败：检测到**错误** → 报错
- 当前设计：检测到**风险** → 中止（不一定是错误）

**结论**：当前设计更像是"风险规避"或"防御性中止"，不是严格的快速失败。

### 9.2 与"乐观锁/悲观锁"的关系

**乐观锁**：
- 假设冲突不会发生
- 执行完再检查，冲突了就回滚

**悲观锁**：
- 假设冲突会发生
- 执行前加锁，阻止冲突

**当前设计**：
- 服务端：悲观（假设新消息 = 可能冲突，中止）
- 端侧（第二版）：乐观锁（版本号校验）

**当前设计取消了端侧乐观锁**，只保留服务端的悲观策略。

### 9.3 与"抢占式/协作式调度"的关系

**抢占式调度**：
- 高优先级任务可以**中断**低优先级任务
- 任务被强制暂停

**协作式调度**：
- 任务主动让出 CPU
- 不会被强制中断

**当前设计**：
- 不是真正的抢占式（工具执行不会被中断）
- 更像是"检查点"设计：在特定点检查，根据情况决定下一步

## 十、总结

### 10.1 演进路径

```
模型一（延迟绑定）
    ↓ 问题：耦合严重
模型二（抢占式 + 乐观锁）
    ↓ 问题：复杂，浪费计算
    ↓ 契机：Agent 性能提升
模型三（信息透传 + Agent 自决）
```

### 10.2 核心设计原则

1. **简单优先**：能简单解决就不要复杂
2. **信任 Agent**：不替它做决策，给它信息让它自己判断
3. **保守策略**：对于不可逆操作，宁可错杀不可放过
4. **性能驱动**：性能条件变了，架构也可以变

### 10.3 当前设计一句话总结

**"不确定就停下来，让 Agent 自己看着办。"**
