## Context

当前仓库没有统一的对象存储模块，业务如果直接接入 MinIO SDK，容易把 endpoint、鉴权、bucket、对象 key 规则和错误处理扩散到多个调用点。用户已经提供了一个可用的 MinIO 服务地址 `115.190.14.209:2012`，因此本次设计聚焦于沉淀一个共享 OSS 模块，让业务只依赖稳定接口，而把 MinIO/S3 兼容协议细节封装在基础设施层。

这个变更属于跨模块基础设施能力：一旦完成，后续媒体、附件、导入导出等场景都可以复用，因此需要在实现前先把接口边界和配置策略定义清楚。

## Goals / Non-Goals

**Goals:**
- 提供一个统一的 `Uploader/Downloader` 或 `ObjectStorage` 抽象，覆盖文件上传和下载。
- 支持 MinIO/S3 兼容 endpoint，优先适配 `115.190.14.209:2012` 这一实例。
- 将 bucket、鉴权、TLS、超时、路径前缀等参数收口到统一配置对象。
- 让业务代码只感知文件流、对象 key 和结果元数据，不直接感知 MinIO SDK 细节。
- 为单元测试保留可替换实现，使调用方可以通过 mock/fake 验证逻辑。

**Non-Goals:**
- 本次不设计复杂的权限系统、预签名 URL、对象生命周期管理或跨 bucket 复制。
- 本次不绑定某一个业务场景，不直接改造现有媒体管线或其他调用方接入。
- 本次不把 MinIO endpoint、bucket 或密钥硬编码进业务代码；默认值可以配置，但最终以运行时配置为准。

## Decisions

### 1. 采用“领域接口 + 基础设施实现”的两层结构

共享模块对上暴露稳定接口，例如 `PutObject`、`GetObject`、`OpenDownloadStream` 等；对下由 MinIO 客户端实现这些接口。这样可以把第三方 SDK 限制在基础设施层，避免后续切换其他 S3 兼容服务时影响业务代码。

备选方案：
- 让业务直接调用 MinIO SDK：实现快，但会把配置、错误模型和对象命名规则扩散到调用方，后续重构成本高。
- 只提供一组 helper 函数：适合简单脚本，不适合作为长期复用模块，也不利于 mock 和依赖注入。

### 2. endpoint 作为配置输入，不在代码中硬编码

虽然当前已知服务地址是 `115.190.14.209:2012`，但模块应通过配置读取 endpoint，并允许配置 scheme/TLS、bucket、region、access key 和 secret key。这样既可以用该地址作为默认部署目标，也能在测试、预发和正式环境中切换实例。

备选方案：
- 将 IP:port 写死在模块内部：对当前环境最省事，但会立刻制造环境耦合和迁移成本。

### 3. 上传与下载统一围绕对象 key 契约，而不是文件路径契约

模块内部以 `bucket + objectKey` 作为唯一资源定位方式。上传接口负责接收 reader/bytes、内容长度、Content-Type 和可选对象 key；若调用方未指定 key，则由模块根据可配置前缀和命名策略生成。下载接口返回对象元数据和流，调用方如需落盘应在上层决定目标路径。

备选方案：
- 以本地文件路径为主接口：会混淆“对象存储资源”和“本地文件系统资源”的边界，不利于流式处理。

### 4. 错误分类在共享模块内统一收口

模块需要把典型错误区分为配置错误、鉴权错误、对象不存在、超时/网络错误和内部 I/O 错误，向上暴露稳定错误语义，而不是把底层 SDK 的原始错误直接泄露给业务。原始错误仍保留在 wrapped error 中，便于日志和诊断。

备选方案：
- 直接透传 SDK 错误：业务侧判断复杂，且未来更换 SDK 时可能破坏调用方逻辑。

### 5. 首版优先支持服务端直连上传/下载，预签名 URL 留作后续扩展

当前需求是“实现 oss 文件的上传和下载”，因此首版以服务端直接执行上传和下载为主，接口设计为将来扩展 `PresignPut`/`PresignGet` 留出口，但不在当前 capability 中承诺。

备选方案：
- 一开始就把预签名、ACL、策略控制全部纳入：能力更全，但会扩大范围，拖慢基础能力落地。

## Risks / Trade-offs

- [运行时配置不完整] → 模块启动时做配置校验，对缺少 endpoint/bucket/credentials 的情况快速失败并返回清晰错误。
- [对象 key 规范不一致] → 在共享模块中定义统一 key 生成策略和可选前缀，避免调用方各自拼接路径。
- [下载接口语义过重] → 默认返回流和元数据，把“是否落盘”留给上层，防止模块同时承担存储和文件系统职责。
- [测试依赖真实 MinIO 环境] → 通过接口抽象 + fake client 保持单元测试可运行；真实联调放到集成测试或环境验证。
- [后续业务直接绕过公共模块] → 在设计和后续代码评审中要求新业务通过共享模块访问对象存储。

## Migration Plan

1. 新增共享 OSS 模块、配置结构和 MinIO 实现，不改动现有业务逻辑。
2. 为模块补齐单元测试，验证配置、上传、下载和错误分类。
3. 选择一个后续调用方试点接入该模块，验证公共接口是否足够稳定。
4. 若出现兼容性问题，可在不影响业务代码的前提下仅回滚模块接入层或切回原调用方式。

## Open Questions

- MinIO 访问凭证由哪套现有配置体系托管，是否已经有统一密钥管理方式？
- 首个 bucket 名称和对象 key 命名规范是否需要按业务维度预留目录前缀？
- 下载接口最终是否需要直接返回本地临时文件能力，还是只保留流式读取即可？
