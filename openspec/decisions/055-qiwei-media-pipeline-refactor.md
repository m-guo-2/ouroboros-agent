# 企微富媒体管线重构方案

- **日期**：2026-03-09
- **类型**：架构决策
- **状态**：已决定

## 背景

当前 `channel-qiwei` 的富媒体处理把消息类型识别、字段兼容、下载接口选择、附件落盘和解析消费混在一条猜测式 fallback 链路里。结果是个微图片/文件会误走企微契约，企微与个微语义混用，同时失败时缺少足够的结构化上下文，难以定位真实根因。

## 决策

将 `channel-qiwei` 的富媒体处理重构为统一媒体管线：先分类、再归一化、再规划下载策略、再执行下载与物化，最后把 agent 可直接消费的结果交给上游。企微与个微视为两套独立文件体系，分别由专属 resolver 执行，不再依赖隐式 fallback 猜测接口。文件类资源必须在 `channel-qiwei` 内部先上传到 OSS，并以“极简占位 + 资源地址”的形式交给 agent；语音则在入口层直接转写为文本，agent 不感知下载或识别细节。

## 变更内容

- 新增中心化的媒体消息分类层，统一维护 `msgType -> source/kind` 映射。
- 新增统一附件描述对象，收口 `fileAesKey/fileAeskey`、`fileAuthKey/fileAuthkey`、`fileBigHttpUrl/fileMiddleHttpUrl/fileHttpUrl` 等字段变体。
- 新增下载规划层，根据 `source + kind + available fields` 显式选择 `/cloud/wxWorkDownload`、`/cloud/cdnWxDownload` 或 `/cloud/wxDownload`。
- 将下载执行器按企微/个微拆分，避免两套契约继续在同一函数里互相污染。
- 统一 `callback` 与 `parse_message` 的媒体处理入口，复用同一条媒体管线。
- 增加资源物化边界：图片、文件、视频统一上传到 OSS。
- 增加语音专用通道：入口层直接转写，agent 只拿文本结果。
- 增加 agent 侧语义收口：`企微/个微` 来源仅在 `channel-qiwei` 内部使用，不进入 agent payload。
- 增加 agent 侧细节收口：agent 只拿到最小必要的 OSS 资源地址用于按需二次解析；MIME、大小、下载策略等技术细节默认不进入 agent payload，文件名也只是可选信息。
- 为媒体处理增加结构化日志、失败分层和回归样本矩阵。

## 考虑过的替代方案

### 继续在现有逻辑上补字段和 fallback

没有采用。这样只能局部缓解个别场景，无法解决“先猜语义、再猜接口”的根本问题，未来新增 `msgType` 或字段变体时仍会持续回归。

### 保留统一大函数，只是把分支再写细一点

没有采用。企微和个微已经是两套不同下载契约，继续堆在同一个函数里只会让规则越来越难验证，也很难为每一层建立清晰测试边界。

## 影响

- `channel-qiwei` 内部会从“松散兼容逻辑”升级为“强语义媒体管线 + 资源准备层”，但 façade 对 agent 的外部接口保持不变。
- 未来支持新富媒体类型时，只需扩展分类表、归一化器和对应 resolver，而不是修改多处调用点。
- 媒体下载失败会从“模糊失败”变成“在哪一层失败、为什么失败”可直接观测，但这些细节只保留在 `channel-qiwei` 内部，不上抛给 agent。
- agent 只看到统一的极简占位、OSS 资源地址和文本，不需要也不会知道原始资源来自企微还是个微，也不需要文件技术细节。
- 后续实现需要补充真实样本或脱敏样本测试，以保证企微/个微两套下载契约长期稳定。
