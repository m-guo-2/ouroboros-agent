# 共享 OSS 模块提案

- **日期**：2026-03-10
- **类型**：架构决策
- **状态**：待实施

## 背景

当前仓库还没有统一的对象存储抽象，如果各业务模块分别接入 MinIO，会把 endpoint、bucket、鉴权、对象 key 规则和错误处理分散到不同位置。现在已经有明确的 MinIO 服务地址 `115.190.14.209:2012`，适合先把上传/下载能力沉淀成一个公共模块。

## 决策

新增一个 MinIO/S3 兼容的共享 OSS 模块，对上提供稳定的上传/下载接口，对下封装具体客户端实现，并把 endpoint、bucket、凭证和前缀等参数统一收口到配置层。

## 变更内容

- 新建 OpenSpec change：`openspec/changes/add-shared-oss-module/`。
- 在 proposal 中确认新增 capability：`shared-oss-storage`。
- 在 design 中明确采用“公共接口 + 基础设施实现”分层，不将 `115.190.14.209:2012` 硬编码进业务逻辑，而是作为配置输入。
- 在 spec 中定义上传、下载、配置校验、统一错误分类和可测试接口这几类行为要求。
- 在 tasks 中拆分为模块骨架、配置接入、上传下载实现和测试接入四类实施任务。

## 考虑过的替代方案

### 让业务直接调用 MinIO SDK

没有采用。这样虽然接入快，但会把第三方 SDK 细节和环境配置扩散到业务代码，后续切换存储服务或统一治理的成本更高。

### 将 MinIO 地址硬编码在模块中

没有采用。当前地址可作为默认部署目标，但硬编码会带来环境耦合，不利于测试、预发和后续迁移。

## 影响

- 后续媒体、附件、导入导出等文件场景可以复用同一套对象存储能力。
- 业务代码对 MinIO 的依赖将被收口到公共模块，错误模型和 key 规则也能统一。
- 真正实施前还需要补齐 bucket、凭证托管和首个接入场景等细节决策。
