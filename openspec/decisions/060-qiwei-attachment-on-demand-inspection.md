# 企微附件按需解析与语音前置转写边界

- **日期**：2026-03-10
- **类型**：讨论结论
- **状态**：已决定

## 背景

`channel-qiwei` 已经把图片、文件、视频物化为稳定资源地址，把语音转成文本，但 agent 侧仍然主要看到“占位文本 + 链接”这一弱协议。真实运行中，模型即使拿到了可解析资源，也可能直接回复“没有视觉能力”，说明问题已经不在下载管线，而在 agent-facing 契约和触发机制。

## 决策

语音继续作为入口层特例前置转写；图片、文件、视频不做前置语义解析，只做资源准备和结构化附件挂载。agent 在需要理解附件内容时，必须通过统一的按需附件分析能力显式调用，而不是从正文里猜测资源地址再拼工具参数。

## 变更内容

- 继续保留 `channel-qiwei` 现有的分类、归一化、下载规划、OSS 物化和语音转写管线。
- 将 `channel-qiwei -> agent` 的入站契约从“单一文本内容”扩展为“文本 + 结构化附件”。
- 为图片、文件、视频引入稳定的 `attachmentId` 与最小附件字段集合，取代正文中的资源地址提示协议。
- 新增统一附件分析能力作为 agent 主流程入口，`wecom_parse_message` 降级为兼容层或桥接层。
- 在 runtime 增加 attachment-aware 约束，确保回答依赖附件内容时不会跳过分析步骤。

## 考虑过的替代方案

### 保持当前正文链接协议，只加强 prompt

没有采用。问题根因是附件被压平成字符串，触发完全依赖模型自行理解协议；继续加 prompt 只会提高命中率，不会让系统边界稳定。

### 对所有富媒体做前置解析

没有采用。语音适合前置转写，因为文本是它的基础可消费形态；图片、文件、视频是否需要理解内容取决于当前问题，默认前置解析会增加时延和成本，也会让入口层承担过多语义工作。

## 影响

- `channel-qiwei` 与 agent 之间的消息协议需要扩展附件结构。
- 多模态主路径会从“字符串提示 + 弱约定工具调用”升级为“结构化附件 + 按需分析”。
- 语音与其他附件的处理边界被明确区分，后续设计和实现不再混用同一策略。
