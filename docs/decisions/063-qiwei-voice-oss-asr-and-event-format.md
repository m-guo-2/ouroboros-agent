# 企微语音改为 OSS 中转识别并统一事件文案

- **日期**：2026-03-10
- **类型**：架构决策 / 代码变更
- **状态**：已实施

## 背景

此前 `channel-qiwei` 的图片、文件、视频已经统一先上传 OSS，再把资源地址交给上游；但语音仍然直接把 QiWe 下载地址提交给火山 ASR。这样一来，语音链路与其他富媒体的资源边界不一致，也无法保证火山拿到的是稳定、可公开访问的地址。

同时，语音转写后下发给 agent 的内容仍然只是裸文本，缺少“是谁发来的语音”这一事件语义，模型侧不容易从上下文中区分这是一条普通文本还是一条转写后的语音消息。

## 决策

语音消息统一改为先上传 OSS，再把 OSS 公网地址提交给火山 ASR；下发给 agent 时，语音消息正文统一格式化为：`xxx(用户)发送了语音消息，转换为文字是：xxxxx`。

## 变更内容

- 在 `channel-qiwei/media_pipeline.go` 中调整语音分支，先复用附件下载/OSS 上传能力，再把返回的公网 URL 交给 `SubmitAudioTranscription`。
- 保持图片、文件、视频与语音的资源准备边界一致，统一由 `channel-qiwei` 负责可访问资源地址。
- 在 `channel-qiwei/events.go` 中为语音消息增加专用事件文案格式，不再沿用通用的 `sender: content` 文本拼接。
- 在 `channel-qiwei/media_pipeline_test.go` 中补充回归用例，验证 ASR 提交地址已经是公网 URL，且 callback 下发给 agent 的语音文案符合预期。

## 考虑过的替代方案

### 继续让语音直接使用 QiWe 下载地址识别

没有采用。这样语音链路会继续成为富媒体管线里的特例，识别服务依赖的地址稳定性也更差。

### 只改事件文案，不改资源准备方式

没有采用。文案只能改善 agent 感知，不解决语音链路仍然绕过 OSS 边界的问题。

## 影响

- 火山 ASR 现在拿到的是由本系统控制的 OSS 公网地址，而不是第三方临时下载地址。
- agent 接收到的语音消息具备明确事件语义，更容易在对话里理解“这是一条转写后的语音”。
- 语音与其他富媒体的资源准备边界进一步统一，后续诊断与扩展更一致。
