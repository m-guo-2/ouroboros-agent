# 企微富媒体向 Agent 暴露资源地址

- **日期**：2026-03-09
- **类型**：讨论结论
- **状态**：已实施

## 背景

上一轮富媒体管线重构把图片、文件、视频统一收口为极简占位文本，避免 agent 感知企微/个微差异和入口层下载细节。这样虽然边界足够干净，但也让 agent 无法在真正需要时继续理解图片或文件内容，只能停留在“我收到了一个资源”这一层。

## 决策

`channel-qiwei` 继续负责下载和资源物化，但对图片、文件、视频会向 agent 暴露“极简占位 + 资源地址”这一最小可行动信息；agent 若要理解内容，必须显式调用 `wecom_parse_message(resourceUri=...)`，旧的 `localPath` 仅作为兼容入参保留。

## 变更内容

- `channel-qiwei` 的媒体 payload builder 从“仅占位”调整为“占位 + 资源地址”。
- `callback` 转发给 agent 时，图片/文件/视频正文中会带上已准备好的资源地址。
- `parse_message` 以 `resourceUri` 作为首选输入，允许 agent 直接解析已物化好的图片或文件。
- 图片解析走现有视觉/OCR provider；文本文件优先本地读取，其他文档再走文档解析能力。
- 语音链路不变，仍由入口层直接转写成文本后交给 agent。
- `企微/个微` 来源、下载策略、失败阶段、MIME、大小等细节仍不进入 agent payload。

## 考虑过的替代方案

### 继续坚持“完全不暴露地址”

没有采用。这样 agent 无法在真正需要时继续理解资源内容，只能依赖入口层预先猜测是否做 OCR/解析，职责边界会重新变得模糊。

### 直接把全部附件元数据都暴露给 agent

没有采用。这样会把入口层内部契约、诊断字段和技术细节重新泄漏到 agent 侧，增加提示词和工具协议复杂度。

## 影响

- agent 对图片/文件的理解从“被动接收入口层预处理结果”改为“按需显式触发解析”。
- `channel-qiwei` 仍然完全屏蔽企微/个微来源差异，只额外提供一个稳定的本地资源入口。
- 当前资源地址已经落到 OSS 语义，只需要保证这个“可解析地址”协议持续稳定即可。
